# WaTerm コード署名ガイド

未署名配布のままだと Windows SmartScreen が初回に「WindowsによってPCが保護されました」警告を出す。
これは実行ファイル自体の問題ではなく、ダウンロード時に付く **Mark-of-the-Web (MOTW)** と
**署名の有無/評価** によるもの。恒久的に消すにはコード署名が要る。

## 選択肢の比較

| 方式 | SmartScreen | 費用 | 備考 |
|---|---|---|---|
| 未署名（現状） | 出る | 無料 | `Unblock-WaTerm.bat` で MOTW 除去は可能だが毎回手間 |
| 自己署名（`build/dist-signed.ps1`） | **消えない** | 無料 | 配布先PCで `.cer` を信頼ルートに入れて初めて有効＝実質不可 |
| OV 証明書（従来型） | 実績を積むと減る | 年 1〜3万円 | 発行に本人/組織確認。評価が貯まるまで数週間出続ける |
| **Azure Trusted Signing** | **即座に通る** | 月 $9.99 前後 | 個人/組織対応。**推奨**。証明書を持たずクラウド署名 |
| EV 証明書 | 即座に通る | 年 3〜10万円＋HSM | 最も強いが高価・ハード必須 |

**推奨: Azure Trusted Signing**（旧 Azure Code Signing）。月額が安く、SmartScreen 評価を最初から引き継げる。

## Azure Trusted Signing を electron-builder に接続する

1. Azure で Trusted Signing アカウントと Certificate Profile を作成（本人確認あり）。
2. 署名用のサービスプリンシパル（App 登録）を作り、`AZURE_TENANT_ID` / `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET` を取得。
3. `package.json` の `build.win` に署名設定を追加：

```jsonc
"win": {
  "signtoolOptions": {
    "publisherName": "あなたの表示名",
    "sign": "./scripts/sign-azure.js"   // カスタム署名フック（下記）
  }
}
```

4. 署名フック `scripts/sign-azure.js` で Trusted Signing の CLI/DLL を呼ぶ
   （`Azure.CodeSigning.Dlib` + `signtool` の `/dlib` 方式）。CI からは環境変数で認証。

## CI での有効化

`.github/workflows/ci.yml` の `build` ジョブに署名ステップの雛形を用意済み。
Repo Secrets に `AZURE_TENANT_ID` / `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET` を登録し、
コメントアウトしている `env:` 行を解禁すれば署名ビルドになる。

## ⚠️ 既存ユーザーへの移行時の注意（重要）

electron-updater は **署名の連続性** を検証する場合がある。今まで **未署名** で配ってきたため、
いきなり署名版に切り替えると、旧版からの自動更新が **署名検証で失敗** する恐れがある。

対策:
- 署名版へ切り替える **最初の1回だけ** は、ユーザーに手動で新インストーラを入れ直してもらう
  （リリースノートに明記）。それ以降の更新は署名の連続性が保たれ自動更新が通る。
- もしくは `latest.yml` の配布と同時に告知し、旧版利用者には案内リンクを出す。

## 現状のまま（署名しない）運用

- 通常の `npm run dist` は **未署名**（既存の自動更新互換を壊さないため）。
- SmartScreen 回避は `Unblock-WaTerm.bat` / `SmartScreen-回避手順.md` を同梱済み。
- 自前ビルドの `release\*.exe` を直接実行する分には MOTW が付かず警告は出ない。
